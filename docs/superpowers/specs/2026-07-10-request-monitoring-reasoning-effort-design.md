# 请求监控思考强度与列对齐设计

## 目标

- 修复请求监控表格中“状态码”表头与状态徽标横向不对齐的问题。
- 新增独立的“思考强度”列，显示最终实际发送给上游提供商的思考强度。
- 将思考强度纳入请求详情和 CSV 导出。
- 不修改现有汇总口径，也不回填无法可靠推断的历史请求。

## 数据口径

运行时 `usage.Record.ReasoningEffort` 已保存翻译后的上游思考强度。用量账本直接持久化该值；仅当记录值为空时，账本插件才回退读取 usage context。

常见值包括 `none`、`low`、`medium`、`high`、`xhigh`、`max` 和 `ultra`。未知但非空的上游值原样保存和显示，避免错误归类。历史记录以及没有思考配置的请求显示 `-`。

## 后端设计

1. 在 `usage_events` 新增 `reasoning_effort TEXT NOT NULL DEFAULT ''`。
2. 通过现有 `ensureUsageEventColumns` 执行增量迁移，不重建数据库或现有 60 天数据。
3. 在 `usageledger.Event` 和 `AnalyticsEventRow` 增加 `ReasoningEffort`。
4. 账本插件从 `usage.Record.ReasoningEffort` 写入事件，并保留 context 回退。
5. 更新事件插入、查询扫描和管理 API JSON 输出。
6. 不把思考强度加入 rollup 主键或汇总表，因为本次只用于单次请求监控，不参与聚合统计。

## 前端设计

请求监控表格改为以下 10 列：

1. 时间 8%
2. 请求 12%
3. 提供商 / 模型 14%
4. 思考强度 7%
5. 认证 / API Key 14%
6. 状态码 8%
7. 延迟 / TTFT 11%
8. Token 用量 14%
9. 费用 7%
10. 操作 5%

表格使用明确的 `colgroup` 列宽，最小宽度提升到 1580px；较窄视口继续横向滚动。思考强度和状态码列的表头、单元格及内部容器全部按整列居中，移除状态内容容器的局部最大宽度，从而消除视觉偏移。

思考强度使用紧凑胶囊显示；空值显示 `-`。请求详情增加同名字段，CSV 增加 `reasoning_effort` 列。

## 兼容与错误处理

- 旧数据库启动时自动新增字段，旧行读取为空字符串。
- 未携带思考配置的请求正常写入空值，不影响请求处理和用量统计。
- 数据库迁移、插入或查询错误继续沿用现有错误路径。
- 新字段为可选 JSON 文本，旧 Management 版本会自然忽略。

- 后端运行相关包测试及 `go test ./...`。
